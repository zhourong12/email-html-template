# 邮件 HTML 模板（润色 + 生成 + 校验 + 审稿）

## 与其它 Skill 的配合顺序

1. **新建或大修本 Skill 正文**时：先按 **create-skills-agent**（`/create-skills-agent`）做多视角设计与对抗审查，再合并进本文件。
2. **要做成可部署产品**（预览站、API、模板库、自动化 lint）：在同一对话或后续会话中启用 **dev-team**，把下文「工程约束」写进架构/验收；本 Skill 作为**质量与流程**单一事实来源（SOT），dev-team 负责实现与部署后浏览器验收。

## 不适用

- 仅修改一句话、不涉及 HTML 与客户端兼容。
- 站内信 / App Push（非 MIME HTML 邮件）。

## 总流程（助手必须按序执行）

| 阶段 | 名称 | 是否可跳过 |
|------|------|------------|
| 1 | 澄清发送场景与占位符约定 | 有歧义时不可跳过 |
| 2 | 润色 → **仅**输出结构化 JSON | 不可跳过 |
| 3 | 由 JSON 渲染邮件 HTML（表格布局 + 内联样式思路） | 不可跳过 |
| 4 | **校验**（检查清单） | 不可跳过；不通过则回到 2 或 3 |
| 5 | **审稿**（二审清单） | 不可跳过；不通过则回到 2 |

## 阶段 1：澄清（最少字段）

向用户或从上下文确认：

- 语言（中/英/混排）。
- **类型**：营销 / 事务通知 / 内部通报（影响退订、语气）。
- **变量占位符**格式（须与阿里云控制台或发信系统一致，例如 `{{name}}` 或 `${name}`）；**禁止**在润色阶段改写占位符字符串。
- 主 CTA 链接是否为变量。
- 图片：是否必须；若必须，是否已有 **HTTPS** 绝对 URL。

## 阶段 2：润色输出契约（JSON only）

模型**不得**直接输出整页 HTML。只输出 **一个** JSON 对象，建议 schema：

```json
{
  "subject": "string, <= 约 78 字符为宜",
  "preheader": "string, 补充主题外信息，避免与正文首段完全重复",
  "sections": [
    { "heading": "string | null", "body": "string (支持 \\n)", "cta": { "label": "string", "url_placeholder": "string" } | null }
  ],
  "footer_notes": ["string"],
  "plain_text_summary": "string，用于生成 text/plain 草稿"
}
```

规则：

- `body` 中保留用户提供的占位符原样；不引入 `<>` HTML。
- `cta.url_placeholder` 写占位符或 `https://...` 真实链接，与阶段 1 约定一致。

## 阶段 3：HTML 生成约束（Gmail + 阿里云直邮）

生成 HTML 时遵守：

- **布局**：外层 `table` 居中，主列宽度约 **600px**；关键结构用 `table`/`tr`/`td`，避免纯 flex/grid 依赖。
- **样式**：以 **内联 `style`** 为主；若使用 `<style>`，仅作补充且须假设 Gmail 可能部分忽略。
- **禁止**：`script`、`iframe`（除非用户明确要求且接受风险）、事件处理器 `on*`。
- **图片**：`img` 必须 `alt`；`src` 为 **https** 绝对地址或占位符；不设依赖外链 CSS 的背景图做唯一信息载体。
- **链接**：`a` 使用内联样式；营销类预留 **退订** 与 **发件身份说明** 的文案位（具体法规由用户侧合规确认）。
- **编码与声明**：`<meta charset="utf-8">`、viewport 可选；避免依赖 Web 字体加载。

推荐工程实现（dev-team 落地时）：MJML 或 React Email 源码 → 构建 → **CSS inlining** → 输出 HTML；本 Skill 在无工具时用手写符合上述约束的 HTML。

## 阶段 4：校验（助手自检，全部通过才可交付）

- [ ] 无 `script` / 无 `on*`。
- [ ] 正文信息不唯一存在于被 Gmail 大量剥离的样式里（关键句在 `td` 文本中可读）。
- [ ] 占位符与阶段 1 约定一致且未被拆断或 HTML 转义破坏。
- [ ] 主 CTA 为可点击 `a`，颜色对比可读。
- [ ] 提供与 HTML 对应的 **纯文本** 版本要点（可由 `plain_text_summary` 扩展为完整 plaintext）。
- [ ] `subject` + `preheader` 无乱码、无多余换行。

未通过：标明失败项 → 回到阶段 2（改文案结构）或阶段 3（改 HTML）。

## 阶段 5：审稿（二审，另一视角）

以「审稿人」身份再过一遍（可与生成用不同语气，但同一助手内二次 pass 即可）：

- **事实与承诺**：是否含未证实数据、绝对化承诺、侵权品牌滥用。
- **语气与受众**：是否与类型（营销/事务）匹配。
- **合规提示位**：营销邮件是否已有退订/理由说明占位（若用户声明为营销）。
- **阿里云侧**：变量是否在 HTML 与控制台模板中可一一对应；是否误用会触发垃圾特征的大图-only 首屏。
- **Gmail**：首屏是否在图片关闭时仍有可读标题与一句说明。

不通过：列出修改项 → 回到阶段 2 或 3。

## 交付物

对用户给出：

1. 最终 **JSON**（润色结果，便于存盘与复用）。
2. **HTML** 全文（可复制进直邮或 SMTP）。
3. **纯文本** 版本。
4. **校验 + 审稿** 各勾选结果摘要（一行表格式即可）。
