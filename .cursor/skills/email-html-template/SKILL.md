# 邮件 HTML 模板（润色 + 生成 + 校验 + 双审核）

## 不适用

- 仅修改一句话、不涉及 HTML 与客户端兼容。
- 站内信 / App Push（非 MIME HTML 邮件）。

## 总流程（助手必须按序执行）

| 阶段 | 名称 | 是否可跳过 |
|------|------|------------|
| 1 | 澄清发送场景与占位符约定 | 有歧义时不可跳过 |
| 2 | 润色 → **仅**输出结构化 JSON | 不可跳过 |
| 3 | 由 JSON 渲染邮件 HTML | 不可跳过 |
| 4 | **模版结构校验**（机械 / 安全 / 兼容清单） | 不可跳过 |
| 5 | **内容审核**（文案与合规） | 不可跳过 |
| 6 | **模版审核**（版式意图、可读性、与 JSON 一致性） | 不可跳过 |
| 7 | 汇总交付 | 不可跳过 |

阶段 4 未通过 → 回到 2 或 3。阶段 5 或 6 未通过 → 标明责任域（文案 vs HTML）后回到 2 或 3。

## 独立审核（子代理 / 并行 Agent）— 何时用、怎么用

**可以。** 内容审核与模版审核依赖**独立上下文**更不容易和「生成者」自我确认偏误；推荐用 **两个并行、只读交付物的 Agent 会话**（或等价能力）完成 5、6 两阶段。

### 建议启用独立审核的条件（满足任一即可）

- 用户明确要求「多轮审核 / 子代理 / 独立审稿」；
- 邮件类型为 **营销、金融、医疗、跨境** 等敏感或高合规场景；
- HTML 较长（例如主模板 **> 8KB**）或多版本对比；
- 助手对阶段 4 自检结果 **不确定**，需要第二意见。

### 调用方式（助手择优）

1. **优先**：**并行**启动两条**独立 Agent 运行**（或同一产品内的「子任务 / 子代理」；具体工具名因环境而异，如部分 IDE 中的 Task、generalPurpose、explore 等，**仅为示例**）：
   - **审核 A — 内容**：只读输入为 **JSON + plain text + subject/preheader**；**不得**改写 HTML。输出固定格式：`PASS|FAIL`、问题列表（严重度 HIGH/MED/LOW）、若 FAIL 则指明回到阶段 2 的哪些字段。
   - **审核 B — 模版**：只读输入为 **HTML 全文 + JSON**（对照用）；**不得**改写文案。输出固定格式：`PASS|FAIL`、问题列表、若 FAIL 则指明回到阶段 3 的具体结构点（标签/样式/占位符）。
2. **无法并行独立会话**时：在同一对话内做 **两次独立 pass**，每次开头声明角色（「仅内容审核」「仅模版审核」），输入完整交付物，输出与上款相同结构的表格；**禁止**用「我觉得差不多」替代勾选清单。

### 独立审核输出合并规则

- 任一侧 **FAIL** 或标 **HIGH** → **整体不交付**，先修复再重跑 4→5→6。
- 两侧 **PASS** 或仅 **LOW** → 主助手汇总阶段 7 交付物。

## 阶段 1：澄清（最少字段）

向用户或从上下文确认：

- 语言（中/英/混排）。
- **类型**：营销 / 事务通知 / 内部通报（影响退订、语气）。
- **变量占位符**格式（须与阿里云控制台或发信系统一致，例如 `{{name}}` 或 `${name}`）；**禁止**在润色阶段改写占位符字符串。
- 主 CTA 链接是否为变量。
- 图片：是否必须；若必须，是否已有 **HTTPS** 绝对 URL。

## 阶段 2：润色输出契约（JSON only）

模型**不得**直接输出整页 HTML。只输出 **一个** JSON 对象，建议 schema：

```json
{
  "subject": "string, <= 约 78 字符为宜",
  "preheader": "string, 补充主题外信息，避免与正文首段完全重复",
  "sections": [
    { "heading": "string | null", "body": "string (支持 \\n)", "cta": { "label": "string", "url_placeholder": "string" } | null }
  ],
  "footer_notes": ["string"],
  "plain_text_summary": "string，用于生成 text/plain 草稿"
}
```

规则：

- `body` 中保留用户提供的占位符原样；不引入 `<>` HTML。
- `cta.url_placeholder` 写占位符或 `https://...` 真实链接，与阶段 1 约定一致。

### JSON 输出失败时的处理（必执行）

- 模型若在 JSON 外包裹 **Markdown 代码围栏**（\`\`\`）或附加说明文字，助手须先 **剥离围栏与前后缀**，再尝试解析。
- **`JSON.parse` / 等价校验失败**（含截断、引号未转义、尾随逗号）：**最多再请求 2 次**重生成，且每次须明确要求：**「只输出一个 JSON 对象，不要 Markdown 围栏，不要解释」**。
- **仍失败**：**停止**进入阶段 3；向用户展示**失败片段**（脱敏后），请用户粘贴可解析的 JSON 或缩小 `sections` 再试；**不得**用「猜出来的」JSON 继续。

## 阶段 3：HTML 生成约束（Gmail + 阿里云直邮）

生成 HTML 时遵守：

- **布局**：外层 `table` 居中，主列宽度约 **600px**；关键结构用 `table`/`tr`/`td`，避免纯 flex/grid 依赖；**表格嵌套宜浅**（见阶段 4 Outlook 项）。
- **样式**：以 **内联 `style`** 为主；若使用 `<style>`，仅作补充且须假设 Gmail 可能部分忽略。
- **禁止**：`script`、`iframe`（除非用户明确要求且接受风险）、事件处理器 `on*`。
- **图片**：`img` 必须 `alt`；`src` 为 **https** 绝对地址或占位符；不设依赖外链 CSS 的背景图做唯一信息载体。
- **链接**：`a` 使用内联样式；营销类预留 **退订** 与 **发件身份说明** 的文案位（具体法规由用户侧合规确认）。
- **编码与声明**：`<meta charset="utf-8">`、viewport 可选；外链字体仅作增强，**inline `font-family` 必须带体面回退栈**（见下节「字体」）。

### 邮件版式与视觉（本 Skill 自包含，不依赖外部 design skill）

生成 HTML 前先定 **Tone**（与品牌/场景一致的一种明确方向），再落笔；下列准则均须在 **邮件技术边界** 内执行（以 `table` + 内联样式为主，慎用复杂动效与大面积渐变）。

**设计意图**

- **用途**：这封邮件要解决什么（通知 / 转化 / 维系）？读者是谁？
- **Tone 示例**（择一并贯彻）：editorial 编辑风、swiss 理性网格、industrial 工业工具感、luxury 克制奢华、brutalist 粗线留白、organic 自然纸色等。
- **记忆点**：整封邮件最好有 **一个** 清晰视觉锚点（色条、字重对比、或排版节奏），避免面面俱到却无重点。

**字体**

- **组合**：**Display（标题层）+ Body（正文层）** 两套栈；`inline style` 里写全 **Google Font 名 + 系统回退**（如 `Georgia, serif` / `Helvetica Neue, Arial, sans-serif`），因部分客户端会忽略 `<link>`。
- **避免**：全信只有 **Arial / 默认系统无衬线** 堆字；为省事全程 **Inter / Roboto**；多封邮件机械复用同一对字体毫无语境。
- **正文字号**：正文优先 **15–17px**，行高约 **1.65–1.8**；过密 **14px** 长段易显廉价。
- **层级**：**小标签**（可配合 `letter-spacing`、全大写）→ **主标题**（22–30px 量级）→ **正文** → **脚注**（11–13px）；同级标题样式勿混用。

**色彩与主题**

- **结构**：**一大块中性底**（纸白/浅灰/浅米）+ **一条主色**（标题/分隔/按钮）+ **少量点缀**（链接、强调句）；忌多种高饱和色块平均分配、互相抢戏。
- **避免**：**白底 + 大紫粉渐变 Hero**、彩虹渐变按钮链等「泛模板」脸；若用紫色，仅作小面积点缀而非整屏。

**版心与间距**

- **节奏**：以 **8px** 为步进（16 / 24 / 32 / 40 / 48）控制 `padding` 与模块间距。
- **留白或密度**：要么 **留白充足、段距明确**，要么 **刻意做信息密度**（如简报表格），不要「既挤又无结构」。

**邮件内可用的「氛围」手段**（替代网页里的动效与复杂纹理）

- 发丝分割线、**左边框/顶色条**、细线框卡片、小圆角（8–12px）、双色按钮；避免把关键信息只放在 **易遭客户端剥离的 background 或渐变** 里。

**动效**

- 邮件中 **默认不做** 依赖 JS/CSS animation 的动效；若有「展开」诉求，用静态分层与标题引导即可。

**应避免的整体观感（与劣质 AI 出稿区分）**

- 纯系统字体 + 灰底白盒 + 无层级；大紫粉渐变头图；组件排列与常见 SaaS 邮件 **完全同质**；正文挤作一团无分段。

**复杂度的取舍**

- **克制型** Tone：靠 **间距、字重、对齐、细线** 体现质感，少色少装饰。
- **张扬型** Tone：可用更强对比色与块面，但仍须保证 **关图可读** 与 **单主色锚点**，且不把信息绑死在渐变背景上。

推荐工程实现（若用构建链路）：MJML 或 React Email 源码 → 构建 → **CSS inlining** → 输出 HTML；本 Skill 在无工具时用手写符合上述约束的 HTML。

## 阶段 4：模版结构校验（机械 + 安全 + 兼容）

主助手（或脚本）逐项勾选，**全部通过**才能进入 5、6：

- [ ] 无 `script` / 无 `on*`。
- [ ] 正文关键信息不唯一存在于易被剥离的样式里（关图后 `td` 内仍有可读标题与一句说明）。
- [ ] 占位符与阶段 1 约定一致，未被拆断、未被错误 HTML 实体化。
- [ ] 每个对外 `a[href]` 为 `https:` 或约定占位符格式；无 `javascript:`。
- [ ] 主 CTA 为 `a`，内联样式可辨；营销/推广类含 **退订或偏好中心** 占位。
- [ ] `subject` / `preheader` 与 JSON 一致且无乱码。
- [ ] 已生成与 HTML 对齐的 **纯文本** 版本。
- [ ] **Outlook / Word 引擎**（Windows 常见）：不假设与 Gmail 像素级一致；避免 **过深 `table` 嵌套**（建议 ≤ 3 层布局意图）与依赖 **VML/条件注释** 才看得懂的唯一信息；关键句仍在 `td` 文本中。
- [ ] （可选）若仓库有 `npm run email:lint` 等命令，有则必跑并贴结果摘要。

未通过：列出编号 → 回阶段 2 或 3。

## 阶段 5：内容审核（文案域）

由 **审核 A**（独立会话）或本对话内「仅内容审稿」pass 执行，只评 **JSON + plaintext + subject/preheader**：

- **事实与夸大**：数据、保证用语、竞品指涉是否越界。
- **合规与敏感**：营销是否含必要提示位；是否含未授权 PII 展示建议；多语言混排是否歧义。
- **语气与受众**：与阶段 1 类型是否一致；文化/节日表述是否得体。
- **与直邮变量**：文案是否可能误改占位符语义（仅检查描述，不检查 HTML）。

输出：`PASS|FAIL` + 问题表（含严重度与修改建议）。**FAIL** → 回阶段 2。

## 阶段 6：模版审核（版式域）

由 **审核 B**（独立会话）或本对话内「仅模版审稿」pass 执行，只评 **HTML +（对照）JSON**：

- **JSON ↔ HTML 一致性**：标题、段落顺序、CTA 文案与链接占位是否与 JSON 一致。
- **邮件客户端风险**：表格嵌套过深、依赖 `linear-gradient` 承载唯一信息、过小字号/对比度；**Outlook** 为弱一致环境，版式以「不裂、可读」为准，不强行对齐 WebKit 预览。
- **设计意图**：是否与既定 Tone 一致；是否落入 skill 所禁止的泛模板脸。
- **垃圾邮件特征倾向**（启发式）：标题全大写堆砌、过多感叹号、首屏大图无 alt 等（不做法务结论，只标风险）。
- **阿里云发信**：变量在 HTML 中可映射；无破坏 MIME 的未闭合标签（目检）。

输出：`PASS|FAIL` + 问题表。**FAIL** → 回阶段 3。

## 阶段 7：交付物

对用户给出：

1. 最终 **JSON**。
2. **HTML** 全文。
3. **纯文本** 全文。
4. **阶段 4** 勾选摘要表。
5. **阶段 5** 内容审核结论（独立会话或本机 pass 输出摘要）。
6. **阶段 6** 模版审核结论（独立会话或本机 pass 输出摘要）。

## 附录：最小 JSON + HTML 骨架（示意）

**JSON（单行字段可展开）：**

```json
{
  "subject": "示例主题",
  "preheader": "补充一行预览",
  "sections": [
    { "heading": null, "body": "正文第一段。", "cta": { "label": "按钮", "url_placeholder": "https://example.com" } }
  ],
  "footer_notes": ["发件方"],
  "plain_text_summary": "纯文本摘要"
}
```

**HTML 骨架（仅结构，须补全内联样式与品牌）：**

```html
<table role="presentation" width="100%" cellpadding="0" cellspacing="0" border="0"><tr><td align="center">
  <table role="presentation" width="600" cellpadding="0" cellspacing="0" border="0"><tr><td>
    <!-- preheader 隐藏 div、标题、正文 td、CTA <a>、退订占位 -->
  </td></tr></table>
</td></tr></table>
```
